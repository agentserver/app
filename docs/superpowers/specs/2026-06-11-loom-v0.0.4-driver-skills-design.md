# Loom v0.0.4 Driver Skills Design

## Goal

Upgrade the Windows installer payload to Loom `v0.0.4` and make the bundled Loom driver usable from Codex Desktop and the minimal VS Code Codex CLI flow without manual skill installation.

## Requirements

- Package Loom `v0.0.4` Windows driver and slave executables.
- Package the Loom `driver-skills.tar.gz` and `driver-codex-prompts.tar.gz` release assets and verify their SHA256 values.
- Install the driver skills into Codex-discoverable user locations during onboarding configuration and on later completed launches.
- Preserve user-owned skill directories and instruction files. The app may overwrite the managed Loom skill copy, but it must not delete unrelated skills.
- Merge the Loom Codex driver prompt into the user's global Codex instructions through a clearly marked managed block.
- Keep the driver MCP server registered in `~/.codex/config.toml` for both Codex Desktop and minimal VS Code modes.
- Add v0.0.4 MCP timeout fields so the driver startup is less brittle.

## Design

The release packaging scripts remain the source of truth for external Loom assets. They will pin `LOOM_RELEASE=v0.0.4`, update the executable hashes, and add cache entries for the two driver support archives. Inno and portable packaging will include those archives next to `driver-agent.exe` and `slave-agent.exe`.

Runtime configuration will get a small `internal/loom` installer helper. The helper will unpack `driver-skills.tar.gz` into both `%USERPROFILE%\.agents\skills` and `%USERPROFILE%\.codex\skills` when those roots can be resolved. Installing into both paths supports current Codex skill discovery and the existing local environment. The helper will also unpack `driver-codex-prompts.tar.gz` and merge `prompts-codex/AGENTS.md` into `%USERPROFILE%\.codex\AGENTS.md` between stable managed markers.

Driver configuration will stay centralized in `loom.WriteDriverConfig`, while Codex MCP registration stays in `codex.UpdateMCPServer`. The MCP table will add `startup_timeout_sec`, `tool_timeout_sec`, and `enabled` fields through the existing `MCPServer` struct so callers do not hand-build TOML.

The launcher and onboarding orchestrator will call the Loom support installer before writing the MCP server entry. This covers first-run onboarding and the completed-console paths for both frontend modes.

## Error Handling

If the driver or slave binary is missing, existing behavior remains unchanged. If support archives are absent, configuration still writes the driver config and MCP server, but no skills or prompt are installed. Archive extraction rejects entries that escape the target directory.

## Testing

Unit tests will cover:

- `codex.UpdateMCPServer` writes timeout and enabled fields.
- `loom.InstallDriverSupport` installs the `multiagent` skill into both user skill roots.
- `loom.InstallDriverSupport` merges and replaces only the managed AGENTS block.
- `ConfigureFrontend` and completed-launch configuration call the support installer and produce driver skills plus MCP config.
- Packaging tests expect v0.0.4 asset paths and the two support archive payloads.

## Out Of Scope

- Changing Loom registration semantics.
- Installing every skill from the Loom repository beyond the v0.0.4 `driver-skills.tar.gz` payload.
- Adding a GUI to manage installed skills.
